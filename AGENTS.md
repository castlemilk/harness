# Agent Operating Rules for Omega Harness

This repository contains the Omega harness and its packages. When acting as an agent inside this codebase, follow these rules to protect the user's work.

## 1. Never destroy uncommitted work without explicit approval

The following git operations are forbidden unless the user explicitly says yes in the current turn:

- `git reset --hard`
- `git clean -fd` or `git clean -fdx`
- force-deleting branches (`git branch -D`)
- force-pushing (`git push --force`)
- any command that discards unstaged or uncommitted changes

If you need to revert, prefer `git checkout -- <file>`, `git restore`, or creating a revert commit.

## 2. Isolate long-running agent work

Agent benchmark runs and self-improvement loops must operate on isolated state:

- Use a git worktree under `.omega/worktrees/` when available.
- Otherwise create a branch named `agent/<task-id>` and commit all agent changes to that branch only.
- Never commit directly to `main` or to the user's current working branch.

## 3. Checkpoint before long-running work

Before starting any of the following, commit or stash the current working tree and tell the user the commit SHA:

- End-to-end benchmark runs longer than 60 seconds.
- Self-improvement loops (`scripts/omega-loop.mjs`).
- Multiple sequential DeepSWE or agent tasks.
- Any automated fix/refactor that touches more than three files.

## 4. Validate before claiming success

- Run `pnpm -r build` and `pnpm lint` after code changes.
- Run the relevant tests or one DeepSWE task before reporting improvement.
- Do not say a task is "done" or "passing" while tests are red or the build is broken.

## 5. Secrets and dependencies

- Do not hardcode API keys, tokens, or credentials in source files.
- Read secrets from `process.env` or the harness provider config.
- Do not add new dependencies without confirming they are needed and actively maintained.

## 6. Async and edge-case discipline

Code generated by or for the harness must handle:

- Promise rejections and `try/catch` propagation.
- Cleanup of timers, event listeners, LSP connections, and servers on shutdown.
- Empty collections, null inputs, and boundary values.
- Explicit validation with clear error messages when the task requires it.

## 7. Prompt changes are experiments

When modifying `packages/agent/src/prompts.ts` or `packages/agent/src/planner.ts`:

- Record the before/after hash in the `PromptVersion` table.
- Validate the new prompt with at least one DeepSWE or E2E run before declaring it better.
- Revert if the new prompt scores lower than the previous baseline.

## 8. If in doubt, ask

When a requested action would violate any rule above, stop and ask the user for explicit approval rather than proceeding.
